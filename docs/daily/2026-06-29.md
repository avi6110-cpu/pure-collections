## Decisions Made

| Decision | Rationale |
|---|---|
| Supabase over Firebase | Relational model fits the data; RLS is correct for multi-tenant isolation; Vault handles token encryption; open source exit path |
| Frankfurt (eu-central-1) | Closest Supabase region to Israel; EU data residency |
| Email/password, confirmation disabled | Internal tool; owner creates accounts; no email server dependency at this stage |
| Three users from day one | Schema already supports it; trivial to add now; avoids mid-pilot scramble |
| `outgoing_email` (not `collections_email`) | Future-proof: the system will eventually send reminders, reports, and quotes — not only collections emails |
| Outgoing email on `tenants`, not `users` | Authentication (who logs in) and communication (where emails come from) are independent concepts. The business has one outgoing address regardless of which user is logged in |
| `tenants.features` jsonb column | Per-tenant feature flags with zero overhead; new capabilities gated behind flags without schema changes |
| `doc_status_key = "customerName\|documentType\|documentNumber"` | Stable join key connecting PURE workflow data to Rivhit documents; identical to what already exists in localStorage — no format change on migration |
| Rivhit token → Supabase Vault | Token gives full read access to the business's financial documents; must never be in localStorage, cookies, API responses, or browser storage of any kind |
| Migration-based workflow | Every schema change in Git before it runs on production; no ad-hoc SQL in the dashboard on production data |
| Secrets entered locally only | Supabase access token and database password are never shared in chat |
| Vault test deferred to Session 3 | The Vault functions are structurally correct; the first real token write in Session 3 serves as the live functional test |

## Lessons Learned

**Verify before every block, not after.** The partial D3 failure (syntax error mid-paste) left the database in an unknown state. The `pg_policies` verification query before attempting cleanup confirmed zero policies existed — we could re-run cleanly. Without that check we might have tried to re-run on partial state and hit duplicate-policy errors.

**Transactional tests are the correct pattern.** A `BEGIN / ... / ROLLBACK` block is the right way to verify a write-path function on a production database. It proves the function works and leaves zero trace. The 4-paste approach with a separate cleanup step is fragile — cleanup can be skipped or fail.

**Column naming is permanent.** `collections_email` would have been wrong the moment the product sent its first non-collections email. The 30 seconds spent on naming saved a future migration on live data.

**Authentication ≠ communication sender.** The clerk's login email and the business's outgoing email address are two different concepts. Conflating them would have created a confusing product: if the clerk account changes, does the outgoing email change too? No. They are independent.

**The user is not a developer.** Every SQL block must be pre-written, tested in reasoning, and ready to paste. Instructions must specify: where to click, what to paste, what result to expect, and what "stop" vs "continue" looks like. This is the correct workflow for this project.

**Migration files must be idempotent.** `CREATE TABLE IF NOT EXISTS`, `CREATE OR REPLACE FUNCTION`, `DROP POLICY IF EXISTS` before `CREATE POLICY` — these patterns ensure a migration is safe to apply on a database that already has the schema. Without them, a re-run fails destructively.
