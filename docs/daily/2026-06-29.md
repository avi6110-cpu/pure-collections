## Lessons Learned

**Verify before every block, not after.** The partial D3 failure (syntax error mid-paste) left the database in an unknown state. The `pg_policies` verification query before attempting cleanup confirmed zero policies existed ‚Äî we could re-run cleanly. Without that check we might have tried to re-run on partial state and hit duplicate-policy errors.

**Transactional tests are the correct pattern.** A `BEGIN / ... / ROLLBACK` block is the right way to verify a write-path function on a production database. It proves the function works and leaves zero trace. The 4-paste approach with a separate cleanup step is fragile ‚Äî cleanup can be skipped or fail.

**Column naming is permanent.** `collections_email` would have been wrong the moment the product sent its first non-collections email. The 30 seconds spent on naming saved a future migration on live data.

**Authentication ‚âÝ communication sender.** The clerk's login email and the business's outgoing email address are two different concepts. Conflating them would have created a confusing product: if the clerk account changes, does the outgoing email change too? No. They are independent.

**The user is not a developer.** Every SQL block must be pre-written, tested in reasoning, and ready to paste. Instructions must specify: where to click, what to paste, what result to expect, and what "stop" vs "continue" looks like. This is the correct workflow for this project.

**Migration files must be idempotent.** `CREATE TABLE IF NOT EXISTS`, `CREATE OR REPLACE FUNCTION`, `DROP POLICY IF EXISTS` before `CREATE POLICY` ‚Äî these patterns ensure a migration is safe to apply on a database that already has the schema. Without them, a re-run fails destructively.

## Decisions Made

| Decision | Rationale |
|---|---|
| Three-client Supabase architecture (browser / server / admin) | Browser for user-scoped RLS writes; server for session-aware SSR; admin (service_role) only for Vault ‚Äî smallest blast radius |
| Token never leaves server | Rivhit token gives read access to all financial documents ‚Äî must be server-only by design, not convention |
| Cloud-first read, localStorage fallback | Cloud is authoritative; localStorage is the rollback path; startup always succeeds even if Supabase is down |
| Dual-write: localStorage first, cloud non-blocking | A failed cloud write shows an ioError banner but never blocks the user's workflow |
| `saveActivity` shared helper | Prevents `handleSaveStatus` and `handleAddActivity` from diverging; single place to add future behaviour (e.g., audit, push) |
| `ON CONFLICT (id) DO NOTHING` for activity | Activity entries are append-only with stable UUIDs ‚Äî no merge needed; idempotency is free |
| Newer-wins in bulk migration (contacts/statuses) | Prevents a stale localStorage snapshot from overwriting a more recent cloud edit made from another device |
| No new migration file for Sessions 2‚Äì3 | Existing schema was complete; no schema gaps found |
